# Koray Research Comparison Report

_Created: 2026-05-24_

## Executive Summary

The template already implements most of the strongest directory SEO tactics from the Gemini/Koray research: clean indexation thresholds, noindex handling for query-modified pages, canonical hubs, listing quality gates, structured data, crawlable hub links, information-gain blocks, freshness audits, analytics hooks, and a template-readiness audit for future copied directories.

The best remaining opportunities are narrower and more practical:

1. Add a reusable topical-map/semantic-brief layer for every copied directory.
2. Improve listing detail pages with stronger question-led EAV summaries.
3. Replace generic "Explore more" and "Similar listings" sections with more contextual bridge sections.
4. Add internal-link and anchor-text governance audits.
5. Keep image/freshness warnings as launch-time checks, not template blockers.
6. Add Search Console and analytics reporting by page type after launch.

Template note: this project is a reusable website template, so missing listing images are not a current implementation priority. Image warnings matter when launching a real live directory with final business data, but they should not distract from improving the template structure itself.

Some Gemini claims should be used as inspiration, not as hard rules. Google does support strong crawl control, canonical clarity, helpful content, structured data, and faceted-navigation management. Google does not publish rules such as "exactly 40 words after every H2," "1:7 hub-spoke ratio," "under 150 links per page," or "maximum 3 repeated anchors." Treat those as optional quality heuristics, not SEO laws.

## Source Reality Check

Google's current public guidance supports several of the template's existing choices:

- Faceted navigation can create large duplicate URL spaces and slow discovery of useful URLs. Google recommends controlling crawl/index exposure for filter URLs. Source: [Managing crawling of faceted navigation URLs](https://developers.google.com/search/docs/crawling-indexing/crawling-managing-faceted-navigation).
- Sitemaps are a way to suggest canonical URLs and signal important pages. Source: [How to specify a canonical URL](https://developers.google.com/search/docs/crawling-indexing/consolidate-duplicate-urls).
- `noindex` prevents indexing only if Google can crawl the page and see the directive. Source: [Block Search indexing with noindex](https://developers.google.com/search/docs/crawling-indexing/block-indexing).
- LocalBusiness structured data can describe business hours, ratings, addresses, and actions when the page is accessible and the markup matches visible content. Source: [Local Business structured data](https://developers.google.com/search/docs/appearance/structured-data/local-business).
- Google recommends people-first content with original information, clear purpose, helpful experience, and transparency about how content is created. Source: [Creating helpful, reliable, people-first content](https://developers.google.com/search/docs/fundamentals/creating-helpful-content).
- Scaled pages created mainly to manipulate rankings can be considered spam, even if generated by automation or AI. Source: [Spam policies for Google web search](https://developers.google.com/search/docs/essentials/spam-policies).

## Already Implemented Well

### Indexation and crawl control

- `src/lib/seo-policy.ts` centralizes route thresholds, high-intent facets, listing quality, noindex behavior, and canonical resolution.
- Query-modified homepage states are noindexed and canonicalized toward cleaner hub URLs in `src/app/page.tsx`.
- `src/app/sitemap.ts` includes only stronger canonical targets: indexable listing details, area hubs, category hubs, neighborhood hubs, area/category hubs, popular-search hubs, and approved high-intent facet hubs.
- Current SEO audit confirms:
  - 3,504 indexable URLs
  - 739 noindex URLs
  - 3,059 indexable listing pages / 128 noindex listing pages
  - 159 indexable area+category hubs / 377 noindex area+category hubs

### Entity and listing quality depth

- Listing quality scoring already includes descriptions, images, contact actions, addresses, geo data, ratings, review counts, hours, categories, service options, proof signals, and Google verification signals.
- Listing detail pages are noindexed when quality is below the threshold or the listing is permanently closed.
- Listing pages already group EAV-style details into services, guest information, transport, nearby places, contact actions, opening hours, reviews, tags, and structured data.

### Structured data

- `src/lib/structured-data.ts` outputs `LocalBusiness`, `BreadcrumbList`, and `ItemList` JSON-LD.
- Listing schema includes address, geo, phone, sameAs links, price range, aggregate rating, and opening hours when available.

### Hub architecture and internal linking

- Homepage SEO links already expose top areas, top cuisines, strong area+cuisine pages, services, dietary needs, and dining styles.
- Hub pages link laterally to related areas, categories, combinations, and useful searches.
- Listing pages link back to area, neighborhood, category, type, dietary, service, and offering pages through `getListingExploreLinks`.

### Information gain

- SEO landing pages already include data-led descriptions, summary stats, price/review signals, best-fit guidance, practical visit notes, FAQs, breadcrumbs, related links, and listing results.
- This is stronger than a thin directory template because pages contain live counts, local signals, service signals, transport signals, and result-specific information.

### Freshness, measurement, and reusable template workflow

- `npm run audit:freshness` already checks hours, images, ratings, reviews, contact actions, status, service options, local context, categories, and quality score.
- `src/lib/directory-analytics.ts` defines page types and high-value actions, and dispatches provider-neutral browser events.
- `npm run audit:template` checks copied-template readiness, including identity, production URL, import report health, listing quality, hub coverage, SEO policy review, and Supabase setup.

## Partially Implemented

### Semantic brief layer

The code has good page models, but there is no explicit "semantic brief" per page type. The Gemini document repeatedly emphasizes one macro context per page, target entities, EAV pairs, question sections, lexical coverage, and allowed link targets.

Practical improvement:

- Add a reusable `directory-semantic-map` or `page-briefs` config that defines each page type's:
  - primary entity
  - search intent
  - required EAV groups
  - allowed supplementary sections
  - preferred internal link targets
  - indexability role

This would help future copies like "Mexican Restaurants in London" avoid drifting into copied Indian restaurant wording or irrelevant pages.

### Question-led EAV summaries

SEO hub pages have FAQs, but the main guide and information-gain blocks are not consistently question-led. Listing detail pages have useful sections, but headings like "Services and features," "Guest information," "Transport," and "Similar listings" are not written as direct searcher questions with compact factual answers.

Practical improvement:

- Add one short answer paragraph at the top of key listing sections:
  - "What services does {Restaurant} offer?"
  - "What dietary options are listed for {Restaurant}?"
  - "Where is {Restaurant} located?"
  - "What transport links are near {Restaurant}?"
- Keep this natural for users. Do not force every section to be exactly 40 words.

### Contextual bridge sections

The site already has "Explore more" and "Similar listings," but they are still fairly generic. The Gemini research's best useful idea here is not the rigid link-ratio claim; it is the concept of explaining why related links are related.

Practical improvement:

- Rename and rewrite listing bridge sections as contextual questions:
  - "Where can I find similar Indian restaurants near {Area}?"
  - "Which related cuisine pages help compare {Restaurant}?"
  - "What service-based searches match this restaurant?"
- Add 1 short sentence before each link group explaining the relationship.
- Keep only the most relevant links instead of up to 24 explore links plus 8 similar listings.

### Internal anchor governance

Anchor text is generated consistently, but there is no audit for repeated anchors, too many page links, weak labels, or links pointing mostly to noindex/query URLs.

Practical improvement:

- Add a report-only audit that renders or models key page types and flags:
  - repeated anchor labels on the same page
  - excessive links in listing detail pages
  - internal links to noindex query URLs from primary content areas
  - generic labels such as "Explore more" where a specific target label is better
  - mismatches between link text and target page intent

### Data provenance and trust explanation

The template has "Updated restaurant details" and freshness audits, but users and Google also benefit from knowing how the directory is built and maintained.

Practical improvement:

- Add a crawlable "How this directory works" or "Data and update policy" page.
- Explain source type, update frequency, what "verified" means, how to report errors, and how freshness checks work.
- Link it from footer and listing pages where trust matters.

### Analytics and Search Console reporting

The event hooks exist, but there is no analytics provider or Search Console reporting workflow yet.

Practical improvement:

- After launch, connect analytics events to a real provider.
- Create a page-type KPI report:
  - homepage
  - listing detail
  - area hub
  - category hub
  - area/category hub
  - neighborhood hub
  - popular-search hub
  - facet hub
- Track impressions, clicks, indexed pages, noindexed pages, detail CTA clicks, saves, shares, compare usage, and pages with poor engagement.

## Missing High-Value Opportunities

### 1. Topical map manifest for copied directories

Create a reusable manifest before copying the template to a new niche. Example fields:

- central entity: "Mexican restaurants"
- source context: "local restaurant comparison directory"
- location: "London"
- core page types: homepage, area hubs, cuisine/style hubs, area+style hubs, service pages, listing pages
- outer page types: dietary, dining style, transport, popular-search pages
- allowed facets: takeaway, delivery, dine-in, outdoor seating, vegan, vegetarian, gluten-free, halal if relevant
- blocked facets: any old Indian-only or restaurant-only facet not supported by the new data

This is the cleanest way to make the template reusable.

### 2. Page-brief generator/checklist

For each route type, define a brief that says:

- what the page is supposed to answer
- what data must appear
- what data makes it indexable
- which related links are allowed
- what structured data should appear
- what must not appear

This turns the Gemini document's "semantic content brief" idea into something practical.

### 3. Listing EAV summary builder

Add a small reusable model that turns raw listing data into compact factual summaries:

- location summary
- review summary
- service summary
- dietary summary
- transport summary
- contact/action summary

These summaries can feed visible page copy, meta descriptions, schema checks, and quality audits.

### 4. Link graph audit

Add an audit command that checks the internal link graph by page type. It should not try to simulate Google's algorithm. It should answer practical questions:

- Are important hubs linked from the homepage?
- Are listing pages linking to the strongest clean hubs?
- Are too many primary links going to parameter URLs?
- Are weak/noindex hubs being overlinked?
- Are copied directories still linking to old niche pages?

### 5. Image enrichment priority workflow for real launches

The freshness audit shows 3,137 of 3,187 listings are missing images. Because this project is a reusable template, this is not a template-quality blocker and should not be treated as a current implementation priority. It becomes important only when the template is copied into a real launch project with final directory data, because images affect trust, user experience, listing quality, share previews, and perceived uniqueness.

Recommended workflow:

- Enrich images first for listings on high-value hubs:
  - top areas
  - top categories
  - area+category pages with enough demand
  - highest-rated / most-reviewed listings
- Keep low-value listings usable but noindex until they have enough useful data.

## Be Careful With These Gemini Claims

Use these as optional QA ideas, not fixed SEO rules:

- "Every H2 must be a question." Useful sometimes, but unnatural if forced everywhere.
- "Every H2 answer must be 40 words." Google explicitly warns against writing to arbitrary word counts.
- "1:7 hub-spoke ratio." There is no official Google requirement for this.
- "Under 150 total links" and "under 15 main-content links." Good as caution against link clutter, but not a universal threshold.
- "Maximum 3 repeated anchors." Repetition can be messy, but the exact number is not a published Google rule.
- "Bypass backlinks through semantic structure." Strong structure helps, but Google still uses many ranking systems, including link-related systems.
- "Algorithmic dominance" and "complete compliance." Treat this as marketing language, not an engineering guarantee.

## Priority Roadmap

### Priority 1: Add a topical-map and page-brief layer

Create a reusable planning/config document or TypeScript config for each directory copy. This gives every future directory a clear central entity, source context, allowed page types, required data groups, and blocked stale-niche terms.

### Priority 2: Improve listing page EAV summaries

Add short factual answer blocks for the major listing sections. Start with location, services, dietary/options, transport, reviews, and contact actions.

### Priority 3: Make related links contextual

Replace generic "Explore more" and "Similar listings" presentation with question-led contextual bridge sections. Reduce link volume and group links by why they matter.

### Priority 4: Add internal-link governance audit

Create a report-only check for anchor repetition, excessive links, links to noindex/query states, weak target labels, and copied-template link drift.

### Priority 5: Keep image gaps out of template implementation scope

Do not prioritize missing images while this remains a template. Keep the audit warning visible for future launch projects, but focus current development on structure, reusable logic, copy safety, internal links, and indexation governance. When the template becomes a real live directory, enrich images and missing categories for high-value hubs first.

### Priority 6: Connect real measurement after deployment

Use the existing analytics hooks and Search Console data to report performance by page type. This is the feedback loop needed to decide which hubs to improve, noindex, merge, or expand.
